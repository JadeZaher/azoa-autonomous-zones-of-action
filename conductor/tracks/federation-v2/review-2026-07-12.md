# Holochain/.NET boundary review

The future Commons integration remains a separate `AZOA.Commons` repository.
The AZOA node owns only `IHoloFederationGateway` and a thin mapping service;
no conductor, Admin API, WebSocket, MessagePack, zome, or Holochain key type
may escape that seam.

## Package and deployment boundary

`Azoa.Commons.Contracts` (`netstandard2.0`) contains versioned entries,
capabilities, errors, and canonical vectors. `Azoa.Commons.Client` (`net8.0`)
owns bounded App-API calls, signals, reconnects, and protocol negotiation.
`Azoa.Holochain.Runtime` (`net8.0`) is optional, owns local conductor lifecycle
and the loopback/UDS-only Admin API, and never flows transitively through the
client.

The conductor agent key is separate from custody/wallet keys. It requires a
canonical reciprocal binding to the L0 node descriptor, including key-rotation
continuity. DNA validity proves format and authorship only; the node still
enforces local peer policy, conformance threshold, exact DNA/schema versions,
and payload attestations fail-closed.

## Delivery gate

Pre-gate work is limited to L0 conformance artifacts, canonical vectors,
offline verification, contracts, package compatibility matrices, and release
tooling. Runtime/client/DHT implementation remains blocked until federation-v2
has three independent production operators and one real cross-node use case.
The known GPL/native-wrapper prior-art client remains prohibited.
