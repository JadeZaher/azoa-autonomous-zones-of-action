# federation-v2 — decision log, 2026-07-10

## Context

An architect review of the prior `spec.md` (2026-07-05 revision) concluded that Holochain
was "cargo-culted" and recommended **dropping it** in favor of signed-JSON-over-HTTPS
(ActivityPub-shaped) federation. The owner **rejects that conclusion** and directs a
**Holochain-first** rewrite.

## Owner direction (verbatim intent)

> "I would rather we were Holochain-first — the hApp IS the public ledger and the community
> can maintain it (a private one too). The federation layer is almost entirely its own
> thing, with node operators likely just getting core functions to do things like submit
> holons for federation and submit audit data for staying certified, while also hosting an
> instance of the hApp in the AZOA deployment or through a separate service if they desire.
> This will likely be its own dedicated base repo, and the deployment will make AZOA
> operators part of a P2P network community. AZOA.Commons probably."

## Decision

1. **Holochain-first, not Holochain-optional.** The community-maintained hApp/DNA is the
   canonical public ledger (a private hApp instance is also allowed). Federation is P2P via
   the Commons DHT. The old "ActivityPub, signed docs over HTTPS" framing is superseded;
   HTTPS is retained only as bootstrap/fallback transport.
2. **Federation is a SEPARATE dedicated repo — `AZOA.Commons`.** It owns the DNA/hApp, the
   runtime, and the Holochain client. The AZOA node monorepo takes only a **thin client**
   (`AZOA.Commons.Client`) + a small mapping service. The old three-package placeholder set
   inside the AZOA monorepo (`AZOA.Holochain.Client/Runtime/HoloFederation`) is replaced by
   "heavy stuff in AZOA.Commons, slim client in the node."
3. **Thin operator surface = 3 core functions:** submit holons for federation, submit
   audit/conformance data to stay certified, host a hApp cell (embedded or separate service).
4. **P2P community, not a hub.** No hub SaaS, no single network owner.
5. **Sovereignty invariant reconciles "no shared DB" with "shared DHT":** the DHT holds
   opted-in **public facts** (references, descriptors, certification proofs) ONLY; private
   SurrealDB state, custody, PII, and unfederated holons never touch the substrate. A shared
   *fact ledger* is not a shared *database*. Value still settles on-chain, never federated.
6. **Deferral discipline preserved.** Activation gate + deferred status unchanged. L0's local
   half remains the alpha-cheap exception.

## Architect findings — kept as VALID even though the drop-Holochain conclusion was rejected

These were genuine and are now captured as prerequisites/open decisions:

- **Conformance manifest has no generation source.** G1–G7 are xUnit pass/fail with no
  machine-readable artifact. → Prereq **P0**: gate-result serializer must exist before
  "submit audit data for certification" is real.
- **Node-signing is a NEW primitive.** `WalletKeyService` does keygen+encrypt only (no node
  `Sign()`). Under Holochain-first this is *mostly moot* — the hApp **agent key** signs
  entries — but we must be explicit about how the agent key ties to the node's existing key
  material and to the L0 descriptor. → Prereq **P2** (identity binding).
- **`HolonType` is unversioned + fail-open + per-node.** Cross-node interpretation needs a
  **versioned type vocabulary**, mismatch fail-closed. → Prereq **P1**.
- **Vendored `NextGenSoftwareUK/holochain-client-csharp` is unusable as a direct dependency**
  (GPLv3-LICENSE-vs-MIT-csproj conflict, native `holochain_serialisation_wrapper.dll`, 3
  external ProjectReferences / no standalone build). → Keep do-not-vendor stance; it's
  prior-art only. `AZOA.Commons` needs its own client story. → Prereq **P3** (open decision).

## Open decisions carried to the future AZOA.Commons build

| # | Decision | Options / notes |
|---|---|---|
| D1 | **Holochain .NET client story** | A: roll-our-own generic .NET conductor client (Admin/App WS, MessagePack, no native DLL). B: Rust binding/sidecar + slim FFI. C: supervised conductor + minimal WS client (pragmatic first step). Runtime lives in AZOA.Commons regardless. |
| D2 | **DNA/schema versioning contract** | How AZOA.Commons DNA/entry schemas version independently of AZOA node releases; the thin client must negotiate/pin a compatible DNA version. |
| D3 | **Node/agent identity binding** | Is the Holochain agent key derived from / bound to the node's existing key material (provably same principal), or independent with a signed linking entry? Node must present a stable, verifiable agent identity tie-able to the L0 descriptor. |
| D4 | **Conformance-manifest generation (P0)** | Shape of the gate-result serializer: which gates, signed how, republish cadence to keep certification live. |
| D5 | **Versioned HolonType vocabulary (P1)** | Where the versioned type contract lives (in Commons? mirrored?) and how cross-node mismatch surfaces fail-closed. |
| D6 | **Public vs private hApp instances** | The community public DNA is canonical; how a private hApp instance interoperates with (or forks from) the public one. |
| D7 | **Embedded vs separate-service hosting** | Lifecycle/ops contract for both hosting modes so the operator's "host a cell" function is uniform either way. |
